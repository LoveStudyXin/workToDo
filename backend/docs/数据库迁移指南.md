# 数据库迁移指南

本项目使用 [Alembic](https://alembic.sqlalchemy.org/) 进行数据库版本管理和迁移。

## 目录结构

```
backend/
├── alembic.ini          # Alembic 配置文件
├── alembic/
│   ├── env.py           # 迁移环境配置
│   ├── script.py.mako   # 迁移脚本模板
│   └── versions/        # 迁移脚本目录
│       └── 5a37d8a94b5e_initial.py  # 初始迁移
```

## 常用命令

所有命令需要在 `backend` 目录下执行。

### 查看当前数据库版本

```bash
python -m alembic current
```

### 查看迁移历史

```bash
python -m alembic history
```

### 创建新的迁移脚本

当你修改了模型文件（`app/models/*.py`）后，运行以下命令自动生成迁移脚本：

```bash
python -m alembic revision --autogenerate -m "描述你的改动"
```

例如：
```bash
python -m alembic revision --autogenerate -m "add avatar field to user"
python -m alembic revision --autogenerate -m "add priority column to todos"
```

### 应用迁移

将所有未应用的迁移应用到数据库：

```bash
python -m alembic upgrade head
```

应用到指定版本：

```bash
python -m alembic upgrade <revision_id>
```

### 回滚迁移

回滚上一个迁移：

```bash
python -m alembic downgrade -1
```

回滚到指定版本：

```bash
python -m alembic downgrade <revision_id>
```

回滚所有迁移（清空数据库结构）：

```bash
python -m alembic downgrade base
```

## 工作流程示例

### 添加新字段

假设你要给 `User` 模型添加一个 `avatar` 字段：

1. **修改模型文件** `app/models/user.py`：

```python
class User(Base):
    __tablename__ = "users"

    # 现有字段...
    avatar: Mapped[Optional[str]] = mapped_column(String(500), nullable=True)  # 新增
```

2. **生成迁移脚本**：

```bash
python -m alembic revision --autogenerate -m "add avatar to user"
```

3. **检查生成的迁移脚本**（在 `alembic/versions/` 目录下），确保内容正确

4. **应用迁移**：

```bash
python -m alembic upgrade head
```

### 添加新表

1. **创建新模型文件**，例如 `app/models/notification.py`

2. **在 `alembic/env.py` 中导入新模型**：

```python
from app.models import user, todo, work_log, report, report_template, notification  # 添加新模型
```

3. **生成并应用迁移**：

```bash
python -m alembic revision --autogenerate -m "add notification table"
python -m alembic upgrade head
```

## 注意事项

### SQLite 的限制

SQLite 对 `ALTER TABLE` 支持有限，本项目已配置 `render_as_batch=True` 来处理这个问题。Alembic 会通过以下步骤处理复杂的表结构变更：

1. 创建临时表
2. 复制数据
3. 删除原表
4. 重命名临时表

### 手动检查迁移脚本

自动生成的迁移脚本可能不完美，建议在应用前检查：

- 确保 `upgrade()` 和 `downgrade()` 函数内容正确
- 检查是否有遗漏的变更
- 确保数据类型正确

### 数据备份

在生产环境应用迁移前，建议先备份数据库：

```bash
cp work_todo.db work_todo.db.backup
```

### 团队协作

- 提交代码时，迁移脚本应该一起提交到版本控制
- 拉取代码后，如果有新的迁移脚本，运行 `alembic upgrade head`
- 避免修改已提交的迁移脚本

## 常见问题

### Q: 迁移脚本生成为空（只有 pass）

**原因**：数据库结构已经和模型一致，没有检测到变更。

**解决**：
- 确保模型文件已保存
- 确保在 `alembic/env.py` 中导入了相关模型

### Q: 迁移失败

**解决步骤**：
1. 查看错误信息
2. 如果是新数据库，可以删除数据库文件重新开始
3. 如果需要保留数据，手动修复迁移脚本或数据库

### Q: 如何重置迁移历史

如果迁移历史混乱，可以重新开始：

```bash
# 1. 删除 alembic/versions/ 下的所有迁移脚本
rm alembic/versions/*.py

# 2. 删除数据库中的 alembic_version 表
# 或删除整个数据库文件

# 3. 重新生成初始迁移
python -m alembic revision --autogenerate -m "initial"

# 4. 应用迁移
python -m alembic upgrade head
```
